ARENA MODE DEBUG REPORT
=======================
Date: 2026-01-06
Project: QuizGambit

1. OBJECTIVE
------------
The goal is to implement "Arena Mode," a multiplayer Jeopardy-style game where:
- A designated "Picker" selects a question from a grid.
- The question opens simultaneously for all players (Sync).
- Players answer within a time limit (e.g., 15s).
- Results (Leaderboard, Speed, Correctness) are shown after everyone answers.
- The winner of the round becomes the next Picker.

2. ARCHITECTURE (Server-Authoritative)
--------------------------------------
To ensure all players see the same state, we moved game logic from the client to the Supabase database.

A. Central State (`lobbies.arena_state`)
   The `lobbies` table now has a JSONB column `arena_state` containing:
   - `phase`: 'PICKING' | 'OPEN' | 'RESULTS'
   - `pickerId`: UUID of the player whose turn it is.
   - `activeQuestion`: Full data of the current question.
   - `revealedQuestions`: Array of IDs of questions already played.
   - `timerEndTime`: Unix timestamp when the round ends.

B. RPC Functions (Server-Side Logic)
   We created PostgreSQL functions to manage state transitions atomically:
   1. `start_arena_session`: Initializes the game, assigns a random picker, sets phase to 'PICKING'.
   2. `open_arena_question`: Sets phase to 'OPEN', records start time, and broadcasts the question.
   3. `submit_arena_answer`: Records a player's answer. IF all players have answered, it auto-transitions to 'RESULTS'.
   4. `next_arena_turn`: Advances the game to the next round ('PICKING'), assigns the winner as the new picker.

C. Client-Side (`ArenaBoard.tsx`)
   - **Realtime**: Subscribes to `lobbies` UPDATE events. When `arena_state` changes, the UI updates instantly.
   - **Polling Fallback**: Every 2 seconds, the client fetches the latest state to self-heal if Realtime packets are dropped.
   - **Optimistic UI**: The client displays timers and buttons based on the server state.

4. CURRENT PROBLEM
------------------
**Issue:** "Ghost Picker" / Input Lock
- Players are reporting they cannot click questions to pick them, even when it is their turn.
- Debugging shows `is_pk: false` (Is Picker = False) on the client side.
- This means `playerId` (Local Auth ID) does not match `arenaState.pickerId` (Server Assigned ID).

**Observations:**
- The debug banner shows `ME: [ID]` and `PICKER: [ID]`.
- Sometimes these IDs appear identical visually but evaluate to `false` in code.
- This suggests a potential issue with:
  a) Whitespace trimming/Hidden characters.
  b) Case sensitivity (UUIDs are sometimes lower/uppercase).
  c) A logic error where the server thinks a *disconnected* player is the picker.

5. DEBUGGING STEPS TAKEN
------------------------
1. **Debug Overlays**: Added a banner to the top of the screen to visualize exact IDs and Phase.
2. **Code Hardening**: Updated ID comparison logic to be case-insensitive and whitespace-trimmed (`cleanId` function).
3. **Ghost Handling**: Updated `nextTurn` logic to default to the *first available player* if the assigned picker is missing.
4. **Manual Override**: Added a red "Force Skip Picker" button for the Host.

6. RECOMMENDATIONS / NEXT STEPS
-------------------------------
1. **Verify ID Source**: Check if `start_arena_session` RPC is grabbing IDs from the `auth.users` table or the public `players` table. Discrepancies here often cause ID mismatches.
2. **Console Logs**: Inspect browser console for the exact string values of `playerId` and `pickerId` during the glitch.
3. **Database Inspection**: Manually check the `lobbies` table in your SQL editor (`SELECT arena_state FROM lobbies WHERE code='YOUR_CODE'`) to see the raw JSON state.
